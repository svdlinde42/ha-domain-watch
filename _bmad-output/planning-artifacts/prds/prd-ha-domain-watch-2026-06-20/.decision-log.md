# Decision Log — Domain Watch PRD

_Created: 2026-06-20 | Author: Simon_

---

## D-001 — Scope: integration, not add-on
**Decision:** Distribute as a HACS custom_component (polling integration), not an HA add-on.
**Reason:** Simpler installation path for end users; no Supervisor dependency. Real-time certstream (websocket) explicitly deferred to v2 add-on.
**Status:** Confirmed by user.

## D-002 — Primary detection source: crt.sh
**Decision:** crt.sh (Certificate Transparency) is the primary signal; dnstwist is secondary/optional.
**Reason:** Nearly all fake webshops acquire a TLS cert, making CT the most reliable early-warning signal. dnstwist catches typo variants but misses "added-word" patterns.
**Status:** Confirmed by user.

## D-003 — RDAP as enrichment only
**Decision:** RDAP data (registrar, registration date, nameservers) enriches detections but is not a detection source itself.
**Reason:** RDAP availability is inconsistent across registrars; using it as enrichment allows graceful degradation.
**Status:** Confirmed by user.

## D-004 — Fast path working mode
**Decision:** PRD drafted in fast path with [ASSUMPTION] tags; user reviews and iterates.
**Reason:** User provided a comprehensive technical spec; fast path is appropriate.
**Status:** Confirmed by user.

## D-005 — dnstwist in manifest.json with lazy import
**Decision:** dnstwist is declared in `manifest.json` requirements (HA installs it automatically at setup) but imported lazily in code (not at module load time).
**Reason:** Omitting it from manifest.json would require manual user installation when enabling the source — unacceptable UX. Lazy import keeps HA startup performance unaffected.
**Status:** Confirmed by user (2026-06-20).

## D-006 — Technical spec routed to addendum
**Decision:** File-level implementation details, phase breakdown, and test strategy go in addendum.md, not the PRD.
**Reason:** PRD covers what and why; implementation spec covers how. Keeping them separate allows downstream UX/architecture work to reference the PRD without noise.
**Status:** Applied.

## D-007 — Store retention: configurable, default 6 months
**Decision:** Persistent store entries expire after a configurable retention period; default is 6 months.
**Reason:** Balances audit trail usefulness against unbounded storage growth on long-running instances.
**Status:** Confirmed by user (2026-06-20).

## D-008 — mark_reviewed reversibility: configurable, default off
**Decision:** Un-marking a reviewed domain is an opt-in capability (disabled by default). When enabled, a separate `unmark_reviewed` service is exposed.
**Reason:** Default-off prevents accidental re-alerting on already-handled domains; opt-in covers the edge case where re-monitoring is intentional.
**Status:** Confirmed by user (2026-06-20).

## D-009 — Binary sensor window: 24 hours
**Decision:** `binary_sensor.domain_watch_new_detection` is `on` if a new detection occurred within the last 24 hours.
**Reason:** Last-cycle-only would be too narrow for long poll intervals (e.g. a 6-hour poll with the automation checked at 5h59m would miss the event).
**Status:** Confirmed by user (2026-06-20).

## D-010 — Reviewed domains exempt from retention purge
**Decision:** Entries with `reviewed: true` in the persistent store are never purged by the retention policy. A reviewed domain never re-alerts, regardless of elapsed time.
**Reason:** Explicit user decision; a domain handled via takedown or accepted as known-safe should not resurface after the retention window expires.
**Status:** Confirmed by user (2026-06-20).

## D-011 — crt.sh cert fields added to event payload
**Decision:** `issuer_name` and `not_before` are included in the `domain_watch_detected` event payload alongside `cert_id`, all three populated only for crt.sh detections (omitted for other sources).
**Reason:** `not_before` distinguishes a new cert on an existing domain from a genuinely new domain registration. `issuer_name` provides context at no cost.
**Status:** Confirmed by user (2026-06-20).

## D-012 — unmark_reviewed service registered unconditionally, gated internally
**Decision:** Both `mark_reviewed` and `unmark_reviewed` are registered on integration load. `unmark_reviewed` returns an error if called while the option is disabled, rather than being dynamically registered/deregistered.
**Reason:** Dynamic service registration in HA is fragile and creates edge cases on option changes mid-run. Unconditional registration with an internal gate is simpler and more predictable.
**Status:** Applied (reviewer finding, 2026-06-20).

## D-014 — Direct notify mechanism: notify.send_message
**Decision:** The coordinator's direct notify path uses the `notify.send_message` action with an `entity_id` parameter, not the legacy `notify.<service_name>` call pattern.
**Reason:** `notify.send_message` is the current HA notification entity API (2024.12.0+). The legacy pattern calls services by name and is deprecated in newer HA versions. The user configures a full entity ID (e.g. `notify.mobile_app_phone`); the coordinator passes it as `entity_id` to `notify.send_message`.
**Status:** Confirmed by user (2026-06-22).

## D-013 — PRD finalized
**Decision:** PRD marked `status: final` on 2026-06-20. All open questions resolved, all reviewer findings applied, no phase-blockers.
**Status:** Complete.
